# 工作区记忆文档

## 工作区定位

本工作区主要用于编程开发、Git 仓库管理、代码修改、脚本编写、问题排查与项目整理。

模型使用可灵活切换，但默认沿用你在其他 Codex 工作区中的协作习惯，并优先保持中文沟通。

## 用户信息

- 艺缤
- 常见任务：代码优化、Bug 排查、脚本编写、仓库初始化、GitHub 推送、配置调整、本地开发环境整理

## 编程与开发偏好

- 先看真实文件、真实配置、真实日志，再下判断，不凭记忆臆测代码状态
- 区分“已验证事实”“推断”“建议”，避免把猜测写成结论
- 结果尽量可直接执行或复用，优先给出实际修改、脚本、文件、命令方案或明确步骤
- 如果请求涉及某个工作区规则、配置文件、源码文件或报错信息，先打开对应实际文件再处理
- 更重视工作流是否真实可落地，而不只是抽象上的最佳实践
- 能直接在本地完成的事情，优先直接落地到文件，而不是只停留在聊天说明
- 默认不要只做分析，若目标明确，尽量直接完成修改、验证、整理和交付

## Git 与仓库协作偏好

- 如果你说“把这个文件夹初始化成 git 项目，并推送到 GitHub”，默认理解为端到端完成初始化、提交、远端配置和推送
- 如果推送失败，不停在报错解释，优先继续排查认证、远端地址、SSH key、分支与权限问题
- 优先使用非交互、可复现的 Git 流程
- 不随意覆盖或回退你已经存在的改动，除非你明确要求
- 若仓库里已有未说明的修改，默认先避让并在必要时说明风险

## 通用协作偏好

- 默认中文回答；若你切换语言，再跟随切换
- 准确、简洁、直接，避免无意义铺垫
- 如果可以做成实际产物，优先生成实际产物，例如 `Markdown`、脚本、配置文件、说明文档
- 推荐或比较方案时，更看重实际适配度、维护成本和复用价值
- 需要联网核验最新信息时，应明确区分本地已知内容与最新外部信息
- 如果你要求节省上下文或提高复用性，优先采用“索引 + 按需展开”的轻量组织方式

## 输入模板

每次尽量按此格式输入，至少写清前四项：

```text
目标：[一句话说明要做什么]
材料：[代码/文件名/报错/配置/背景]
约束：[格式/范围/不能动什么/是否需要联网]
输出：[期望结果，如直接修改文件、给修复方案、生成脚本]
补充：[可选]
```

## 场景索引

- 通用：说明文档、资料整理、结构化归纳、轻量问答
- 代码：代码修改、调试、重构、脚本编写、报错定位
- Git：仓库初始化、提交整理、分支处理、远端配置、推送排错
- 短任务：追求快速完成、低上下文消耗、一次性小修改

## 对话处理原则

- 新任务开始时，先判断它更像通用、代码、Git 还是短任务
- 如果信息不足但仍可安全推进，先根据现有文件继续做，并在结果里说明采用了什么假设
- 如果会影响现有代码、历史提交、远端仓库或认证配置，应先确认风险点再继续
- 若你明确要求保存为本地文件，应直接创建或更新对应文件

## 机器与环境提示

- 当前机器为 Apple Silicon Mac，涉及安装方法、命令行工具、开发环境建议时，优先考虑 macOS 与 arm64 原生兼容方案
- Shell 默认为 `zsh`
